# Правила проекта

> Короткие и практические правила для этого проекта. Загружаются автоматически командой $aif-implement.

## Rules

- Каждое изменение обязано оставлять зелёными `npm run typesCheck`, `npm run lint` и `npm test`; код с красными quality gates не считается готовым.
- Любое изменение публичного поведения CLI или MCP, включая `inputSchema`, формат ответа и пограничные сообщения об ошибках, обязано сопровождаться обновлением документации и тестов.
- Любое новое поле конфигурации обязано быть добавлено одновременно в `src/config/schema.ts`, `src/config/defaults.ts`, `docs/configuration.md` и тесты конфигурации.
- Модули должны импортировать другие модули только через их публичный `index.ts`; прямые импорты внутренних файлов чужого модуля запрещены.
- Модули `commands/` и `mcp/` должны оставаться тонкими адаптерами и не должны содержать бизнес-логику поиска, индексации, chunking или работы с провайдерами.
- Новые провайдеры и внешние интеграции должны подключаться только через интерфейсы и factory-функции; orchestration-код не должен зависеть от конкретной реализации.
- Ошибки на границах системы должны содержать контекст операции и ключевого идентификатора; безликие сообщения вроде `Error occurred` и `Something went wrong` запрещены.
- CLI-команды должны писать штатный результат только в `stdout`, ошибки только в `stderr`, а MCP-сервер не должен писать в `stdout` ничего, кроме протокола.
- Каждый pull request обязан проходить CI с `npm run typesCheck`, `npm run lint` и `npm test`; ослаблять pipeline без эквивалентной проверки качества запрещено.
- Любое изменение логики `search`, `indexer`, `chunks`, `export`, `import` или MCP-инструментов обязано сопровождаться regression-тестом; bugfix без теста допустим только если тест технически невозможен и причина явно зафиксирована.
- Новая функциональность не должна снижать покрытие критичных модулей `config`, `search`, `indexer`, `mcp` и `storage`; при исправлении бага сначала добавляется или обновляется тест, воспроизводящий дефект.
- Любое изменение схемы БД обязано оформляться отдельной миграцией в `src/storage/migrations/`, быть идемпотентным и сопровождаться обновлением тестов и документации там, где меняется поведение.
- Любое изменение SQL-запросов, влияющее на фильтры, сортировку или производительность поиска, обязано сопровождаться проверкой индексов и тестом на ожидаемое поведение.
