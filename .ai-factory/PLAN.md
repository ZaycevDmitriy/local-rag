# План исправления несоответствий RULES.md

## Контекст

Аудит проекта по `.ai-factory/RULES.md` выявил несколько системных несоответствий:

- отсутствует CI-пайплайн с обязательными quality gates;
- отсутствует coverage gate для Vitest и нет отдельного сценария проверки покрытия;
- часть модулей импортирует внутренние файлы других модулей напрямую вместо публичных `index.ts`;
- документация и AI-контекст по конфигу отстают от фактической схемы;
- в публичной схеме конфигурации объявлен `self-hosted` provider, который не реализован;
- часть миграций БД не является идемпотентной;
- часть CLI-команд содержит избыточную orchestration-логику и generic boundary errors.

## Цель

Привести проект к практическому соответствию правилам качества, модульных границ и инфраструктурной дисциплины, не меняя пользовательское поведение без необходимости.

## Задачи

- [x] 1. Привести межмодульные импорты к публичным barrel API.
  - Обновить импорты `config`, `storage`, `search`, `embeddings`, `sources`, `chunks` так, чтобы внешние модули использовали только `index.ts`.
  - Добавить недостающие barrel-экспорты, где это нужно для соблюдения правила.

- [x] 2. Утоньшить CLI/MCP-адаптеры и вынести orchestration-логику в профильные модули.
  - Вынести orchestration индексации из `src/commands/index-cmd.ts` в `src/indexer/`.
  - Вынести расчёт системного статуса в переиспользуемый модуль вне `commands/` и `mcp/tools/`.
  - Устранить generic boundary errors в CLI-командах, добавив контекст операции.

- [x] 3. Синхронизировать публичный конфиг, документацию и AI-контекст.
  - Удалить из runtime-конфига неподдерживаемый `self-hosted` provider.

- [x] 4. Добавить CI и coverage gate.

  - Настроить пороги покрытия и включить их в GitHub Actions workflow вместе с `typesCheck`, `lint`, `test`.

- [x] 5. Сделать миграции БД идемпотентными и согласованными с правилами.
  - Исправить существующие миграции, где отсутствует `IF NOT EXISTS`/безопасный повторный запуск.
  - Убедиться, что изменения схемы и индексов не ломают повторное применение.

- [x] 6. Прогнать и зафиксировать verification.
  - Запустить `npm run typesCheck`, `npm run lint`, `npm test`.

  - Убедиться, что новые правила реально соблюдаются кодом и инфраструктурой.
