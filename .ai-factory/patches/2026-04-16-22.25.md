# Console.log в MCP-reachable коде ломал JSON-RPC stdio

**Date:** 2026-04-16 22:25
**Files:** src/embeddings/openai.ts, src/search/coordinator.ts, src/storage/chunk-contents.ts, src/storage/chunks.ts, src/mcp/tools/read-source.ts, src/mcp/__tests__/stdout-discipline.test.ts
**Severity:** high

## Problem

Внешний код-ревью на PR #6 (`#issuecomment-4262746902`) указал, что `console.log` на success-path `OpenAITextEmbedder.callApi` (`src/embeddings/openai.ts:154`) пишет в `stdout` при каждом успешном запросе эмбеддингов. Так как эмбеддинг вызывается из `SearchCoordinator` во время обработки MCP `search` tool, каждая такая строка попадает в поток, который MCP зарезервировал под JSON-RPC — и ломает протокол у клиентов (Claude Code, Cursor) при использовании OpenAI-совместимых провайдеров.

При проверке выяснилось, что это не единичный случай: аналогичные `console.log` обнаружились в `SearchCoordinator` (4 сайта), `ChunkContentStorage.searchBm25/searchVector` (2), `ChunkStorage.getContentHashes/resolveOccurrences` (2), `read_source` tool (7). Итого 16 сайтов в MCP read-path — любой успешный `search` или `read_source` фактически выдавал на stdout посторонние строки.

## Root Cause

Архитектурно проект запускает CLI и MCP-сервер из одних и тех же модулей, но контракт `stdout` у них разный:

- CLI: `stdout` = штатный результат, `stderr` = диагностика/ошибки.
- MCP: `stdout` = только JSON-RPC; всё остальное строго в `stderr` (зафиксировано в `.ai-factory/RULES.md` #14 и в заголовке `src/mcp-entry.ts`).

При написании `search`/`storage`/`embeddings` модулей отладочные строки ставились как `console.log`, потому что:

1. В момент написания методов основной сценарий был CLI (`rag index`, `rag status`), где `console.log` выглядит нормально.
2. MCP path появился/расширялся позже, но ревью не подсветило, что те же методы теперь вызываются из MCP-обработчиков.
3. Явного механизма (logger-абстракция, ESLint-правило, тест-guard) для защиты stdout в MCP-коде не было — нарушение ловилось только по факту битого протокола у клиентов.

Соседние сайты в `openai.ts` (строки 120/131/143) уже использовали `console.error` — паттерн в коде был, но не соблюдался системно.

## Solution

1. Во всех 16 MCP-reachable success-path сайтах заменил `console.log` → `console.error`:
   - `src/embeddings/openai.ts:154` — success ветка `callApi`.
   - `src/search/coordinator.ts:70, 84, 221, 233` — оркестрация branch-aware поиска.
   - `src/storage/chunk-contents.ts:125, 162` — `searchBm25`, `searchVector`.
   - `src/storage/chunks.ts:186, 215` — `getContentHashes`, `resolveOccurrences`.
   - `src/mcp/tools/read-source.ts:97, 106, 144, 201, 260, 275, 277` — все ветки tool-обработчика.
2. Оставил `console.log` нетронутым в CLI-only путях (`src/commands/*`, `src/indexer/*`, `src/sources/git.ts`, `src/sources/fingerprint.ts`, migrations) — там это корректный канал для пользовательского вывода.
3. Добавил статический тест-guard `src/mcp/__tests__/stdout-discipline.test.ts`, который читает MCP-reachable файлы и валит сборку, если `console.log(` хоть где-то проявится. Для shared-файлов (`storage/chunk-contents.ts`, `storage/chunks.ts`) проверка ограничена телом конкретных методов, чтобы не ломать CLI-ориентированные методы в том же файле.

Подход `console.error` выбран вместо создания logger-абстракции, чтобы не расширять scope фикса: соседние error-ветки уже используют эту конвенцию.

## Prevention

- При добавлении отладочного логирования в модули под `src/search/**`, `src/mcp/**`, `src/storage/**`, `src/embeddings/**` **сразу** использовать `console.error`. `console.log` в этих директориях = баг.
- Новый guard-тест (`src/mcp/__tests__/stdout-discipline.test.ts`) автоматически ловит регрессии — при добавлении MCP-достижимого файла/метода обновлять списки `MCP_REACHABLE_FILES` / `MCP_REACHABLE_METHODS`.
- На код-ревью в PR, затрагивающих shared-модули: явно проверять call graph — «этот метод вызывается из `SearchCoordinator.search` или из MCP-tool?». Если да — `console.log` запрещён.
- Долгосрочно стоит рассмотреть ESLint-правило `no-console` с исключениями для `console.error` в директориях `src/search`, `src/mcp`, `src/storage`, `src/embeddings` — превратит правило из runtime-проверки в compile-time. Для этой задачи не делал, чтобы сохранить минимальный скоуп.
