## Problem

Ревью фазы 11 (AI-powered summarization) выделило семь не-блокирующих, но системных
замечаний, накопленных в PR #11:

1. **#3** — `summarize-cmd.ts` стал толстым адаптером: dry-run оценка (≈30 строк) и
   while-loop (≈40 строк) нарушали `RULES.md` правило «commands/ — тонкие адаптеры»
   и не были unit-testable без моков Commander.
2. **#4** — `AVG_TOKENS_PER_CHUNK = 200` и `PRICE_PER_TOKEN = 0.05/1M` захардкожены
   под Qwen2.5-7B. Смена `summarization.model` в конфиге молча оставляла dry-run с
   ценой Qwen — ошибка никак не сигнализировалась пользователю.
3. **#5** — `SearchCoordinator` делал `hasSummaryForViews` (SQL EXISTS + JOIN по
   views) на **каждом** MCP `search` при включённом `useSummaryVector`. Лишний
   round-trip на горячем пути.
4. **#6** — `summarization.concurrency: z.number().int().positive()` без верхней
   границы. Пользователь мог задать `1000` и мгновенно упереться в rate-limit
   SiliconFlow.
5. **#7** — `detectDocstring` эвристика ловила только JSDoc/JavaDoc/Kotlin/`"""`, но
   называлась нейтрально. При будущем Python-extractor она молча вернёт `false` для
   `'''` docstring → LLM получит неверный флаг `hasDocstring`.
6. **#8** — В `chunk-contents.ts` одни методы использовали `console.error` (после
   фикса 2026-04-16-22.25), другие (`insertBatch`, `updateEmbeddings`,
   `deleteOrphans`) — оставались на `console.log`. Разнородность опасна при любом
   будущем MCP-reachable добавлении.
7. **#9** — Миграции `005_branch_views_rebuild` и `006_summarization` писали в
   `console.log`. При запуске `rag init` внутри orchestrator'а stdout забивался
   информационными сообщениями.

## Root Cause

- Адаптер распух в момент, когда в фазе 11 добавляли cost-estimate и keyset-loop
  рядом с уже существующей .action() — helper был написан только для батчевой
  обработки (`processSummarizeBatch`), но не для dry-run и main loop.
- Cost-константы рождались как локальные `const` на этапе PoC и не мигрировали в
  конфиг при стабилизации фичи.
- Кэш `hasSummaryForViews` не был заложен сразу, потому что функция казалась
  дешёвой (один EXISTS); реальные замеры на MCP-пути показали, что любой лишний
  SQL на hot-path ощутим.
- Zod-максимумы не обсуждались в design-doc суммаризации — схема делалась по
  минимуму, без edge-case защит.
- Имя `detectDocstring` отражало намерение, а не реализацию — типичная ошибка
  naming-by-intent.
- `console.log` vs `console.error` унификация делалась точечно под конкретные
  MCP-reachable методы, а не полностью по файлам / классам — поэтому каждый новый
  метод приходилось вручную сверять с дисциплиной.

## Solution

### #3 — Тонкий адаптер (commands/)

Выделены три helper'а в `src/commands/_helpers/`:

- `summarize-input.ts` — `toSummarizerInput` + `detectJsStyleDocstring`.
- `summarize-dry-run.ts` — `estimateDryRun` (чистая функция без I/O) +
  `formatCost`.
- `summarize-run.ts` — `runSummarizeLoop` с контрактом `SummarizeRunStorage =
  Pick<ChunkContentStorage, 'getWithNullSummaryForSource' |
  'updateSummaryWithEmbedding'>`, `onProgress` callback для презентации.

`summarize-cmd.ts` ужался с 222 до ~150 строк и теперь только парсит опции,
резолвит зависимости и делегирует. Unit-тесты
`summarize-dry-run.test.ts` (22 кейса) и `summarize-run.test.ts` (4 кейса)
покрывают бизнес-логику без моков Commander.

### #4 — Cost-параметры в конфиге

`SummarizationCostSchema` получил поля `avgTokensPerChunk` (default 200) и
`pricePerTokenUsd` (default `0.05/1M` = Qwen2.5-7B). Dry-run вывод теперь
явно печатает текущие `model`, `avgTokensPerChunk`, `pricePerTokenUsd` — смена
модели без обновления цены становится видимой.

Изменения отражены в `defaults.ts`, `docs/configuration.md`, `loader.test.ts`,
а также в `summarize/__tests__/factory.test.ts` (там фиксируется весь
`AppConfig`).

### #5 — TTL-кэш `hasSummaryForViews`

В `SearchCoordinator` добавлен `summaryCache: Map<string, { result: boolean;
at: number }>` с TTL 5 мин (= `SOURCE_CACHE_TTL_MS`). Ключ — отсортированный
join sourceViewIds через `|`. FIFO-эвикция при превышении
`SUMMARY_CACHE_MAX_ENTRIES = 128`. Два новых теста в `coordinator.test.ts`:
(1) повторные вызовы с одним и тем же набором views делают только один SQL;
(2) порядок sourceViewIds не ломает кэш.

### #6 — `concurrency .max(64)`

`concurrency: z.number().int().positive().max(MAX_SUMMARIZATION_CONCURRENCY)`.
Два unit-теста: 1000 → Zod бросает, 64 → валидно. Docs и дефолт обновлены.

### #7 — Переименование детектора docstring

`detectDocstring` → `detectJsStyleDocstring`. Имя отражает реализацию: ловит
JSDoc/JavaDoc/Doxygen/`"""`, **не** ловит Python `'''` и Ruby `=begin`. Тесты
явно фиксируют лимитации — будущий Python-extractor не сможет молча проехать.

### #8 — Унификация stdout в `chunk-contents.ts`

Все `console.log` (в `insertBatch`, `updateEmbeddings`, `deleteOrphans`)
заменены на `console.error`. Файл повышен из `MCP_REACHABLE_METHODS` в
`MCP_REACHABLE_FILES` в `stdout-discipline.test.ts` — теперь любой будущий
метод автоматически под охраной guard'а, без ручного обновления списка.

### #9 — Миграции 005/006 и regression-guard

`console.log('[migration:...')` → `console.error` в обоих файлах.
Добавлен `describe('storage/migrations: вспомогательные логи через stderr')`
в `stdout-discipline.test.ts`, который динамически сканирует
`src/storage/migrations/NNN_*.ts` и требует отсутствия `console.log`. Любая
новая миграция автоматически попадает под проверку.

## Prevention

- **Тонкие адаптеры как дизайн-цель**: при первом же добавлении >20 строк
  бизнес-логики в `.action()` Commander'а выделять helper. Правило из `RULES.md`
  работает только если применяется в момент роста, не постфактум.
- **Cost/price константы всегда через конфиг**: если код зависит от ценника или
  лимитов провайдера, значение обязано быть в схеме и печататься в dry-run выводе.
- **Кэши на горячих путях по умолчанию**: любой SQL/round-trip, исполняемый на
  каждом MCP-инструменте, требует обоснования «почему без кэша». Сейчас в
  `SearchCoordinator` два TTL-кэша (`sourceCache`, `summaryCache`) — шаблон
  должен применяться к новым EXISTS-запросам тоже.
- **Zod-максимумы для пользовательских лимитов**: при добавлении числовых полей,
  которые ограничиваются здравым смыслом (`concurrency`, `timeoutMs`, `batchSize`),
  всегда задавать верхнюю границу в схеме.
- **Имена функций — название реализации, не намерения**: `detectDocstring` vs
  `detectJsStyleDocstring` — разница в том, что второе имя сразу сигнализирует
  пользователю о лимитациях.
- **Stdout-дисциплина на уровне файла/класса, не метода**: когда файл попадает
  в MCP call-graph хотя бы частично, безопаснее унифицировать весь файл и
  добавить его в `MCP_REACHABLE_FILES`. Список методов — более хрупкая защита.
- **Динамические guard'ы вместо явных списков**: `stdout-discipline.test.ts` для
  миграций сканирует директорию → новая миграция `007_*.ts` автоматически под
  проверкой. Такой pattern стоит тиражировать на будущие директории с единой
  политикой.
