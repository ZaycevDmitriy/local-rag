**Severity:** high (critical регрессия для siliconflow users + suggestion-баги)

## Problem

По итогам `/aif-review PR #11` (AI-powered summarization) нашли один Critical и
два Suggestion-бага:

1. **Critical:** `src/commands/init.ts:30-33` выбирал dimensions по факту наличия
   под-объекта (`config.embeddings.jina?.dimensions ?? openai?.dimensions ?? 1024`),
   игнорируя `config.embeddings.provider`. При `provider: siliconflow` + кастомной
   dimensions (например, будущий Qwen3-Embedding-8B = 4096) миграции создают
   `vector(1024)` на обоих колонках (`chunk_contents.embedding` и
   `chunk_contents.summary_embedding`), embedder возвращает 4096, и **оба** INSERT-канала
   падают с pgvector dimension mismatch.
2. **Suggestion:** `getWithNullSummaryForSource` в `src/storage/chunk-contents.ts`
   использовал `DISTINCT ON (cc.content_hash) ... ORDER BY cc.content_hash` без
   вторичного тай-брейкера. PostgreSQL выбирает «первую попавшуюся» строку по
   physical row order — representative (path/language/metadata) для одного
   content_hash мог меняться между прогонами `rag summarize` → prompt LLM
   нестабилен → бенчмарк не воспроизводим.
3. **Minor:** dry-run sample использует первые 500 записей по `ORDER BY content_hash`.
   Без комментария читатель мог принять это за баг и заменить на `ORDER BY RANDOM()`,
   потеряв детерминированность.
4. **Minor:** спека skip-gate не фиксировала поведение для TYPE/INTERFACE **с** docstring
   (gate пропускает только те, у которых docstring отсутствует).

## Root Cause

1. **Скрытая связь «provider ↔ dimensions» была дважды нарушена:**
   - Изначально резолв в `init.ts` был cheap ternary по наличию объекта — при
     двух провайдерах (jina/openai) работало, потому что их dimensions совпадали с
     их дефолтами. Когда добавился третий провайдер (`siliconflow`), ветвь просто
     не была добавлена — `provider` не участвовал в решении вообще. PR #11 ничего
     не сломал, но тиражировал проблему на вторую vector-колонку (`summary_embedding`),
     увеличив blast radius.
   - Мораль: когда размерность схемы зависит от нескольких независимых источников
     конфигурации, резолв должен быть **одной чистой функцией**, привязанной к
     верхнеуровневому discriminator-полю (`provider`), а не строчкой nullish-коалесценций.
2. **DISTINCT ON без полного тай-брейкера — классическая ловушка PostgreSQL.**
   При JOIN на many-to-one (`chunks → chunk_contents` через `content_hash`)
   и отсутствии уникального вторичного ключа в `ORDER BY`, планировщик волен
   менять порядок строк от прогона к прогону (например, после VACUUM, после
   INSERT новых occurrences, после смены плана из-за статистик). Тест «прогон
   прошёл стабильно локально» ничего не доказывает.
3. **Поведенческий контракт skip-gates был недоспецифицирован.** Код
   (`src/summarize/gates.ts:30`) и тест покрывали случай «TYPE без docstring → skip»,
   но спека не делала явной второй ветки — это типичный drift «код описывает себя,
   а документация — нет», который всплывает при онбординге и ревью.

## Solution

1. **Ввёл `resolveEmbeddingDimensions(config: EmbeddingsConfig): number`** в
   `src/embeddings/dimensions.ts` — switch по `config.provider`, fallback на
   `DEFAULT_EMBEDDING_DIMENSIONS = 1024` (для неуказанного/неизвестного провайдера).
   Использую в `src/commands/init.ts` с explicit console.log про resolved
   provider+dimensions (для диагностики будущих mismatch-инцидентов).
2. **Добавил вторичный `ORDER BY c.id`** в `getWithNullSummaryForSource`
   (`c.id` — уникальный PK) + комментарий-якорь, объясняющий, зачем он.
   Bench-стоимость: сортировка 50 строк по второму ключу ≈ 0% (DISTINCT ON и
   так требует ORDER BY).
3. **Комментарий в `summarize-cmd.ts`** над вычислением `sampleSize`: явно пишу,
   что это детерминированный sample по content_hash (SHA-256 равномерен), а не
   баг, который просит замены на `RANDOM()`.
4. **Обновил `docs/specs/ai-powered-summarization.md:86`** — разложил ветку
   gate-2 в bullet-лист: `docstring есть → summarize`, `docstring нет → skip`,
   c пояснением маркеров.

## Prevention

- **Discriminator-тесты для конфига:** любая функция, которая резолвит что-то
  из discriminated-union конфигурации (`provider: 'x' | 'y' | 'z'`), должна
  иметь матричный тест «каждый provider × дефолтные vs кастомные значения» —
  7-8 кейсов минимум. Это единственный барьер, который ловит «forgot to add new
  branch when provider added» до продакшна.
- **DISTINCT ON code-review checklist:** в `ORDER BY` должен быть хотя бы один
  столбец с `UNIQUE` / `PK` constraint после DISTINCT-ключей. Иначе запрос
  недетерминирован по определению. Добавить это в `.ai-factory/skill-context/aif-review/SKILL.md`
  как pattern на будущее.
- **Spec ↔ code sync:** если gate (или любая условная ветка) упоминается в спеке,
  проверить, что в спеке описан ВЕСь исход (включая ветку «не сработал»), не
  только «срабатывает при X». Это защищает от молчаливой смены поведения под
  видом «уточнения».
- **Fail loudly при provider/embedder dimension mismatch:** отдельный runtime-check
  на старте CLI (`rag index`, `rag re-embed`, `rag summarize`), сравнивающий
  `embedder.dimensions` с фактической размерностью колонки `chunk_contents.embedding`
  (`SELECT atttypmod - 4 FROM pg_attribute ...`) — это отдельный фичреквест, но
  сейчас `resolveEmbeddingDimensions` закрывает только init-канал.
