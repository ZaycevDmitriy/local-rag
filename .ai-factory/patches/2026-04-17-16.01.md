# Atomic batch updates + failed-row placeholders in LLM backfill loop

**Date:** 2026-04-17 16:01
**Files:** src/storage/chunk-contents.ts, src/commands/summarize-cmd.ts, src/commands/_helpers/summarize-batch.ts
**Severity:** critical

## Problem

Команда `rag summarize` в PR #11 содержала два дефекта, проявлявшиеся на первом же
продовом прогоне:

1. Retry-loop: при persistent LLM-failures (HTTP 400, timeout, OOM на конкретных чанках)
   failed rows НЕ помечались в БД. Внешний `while (processed < maxToProcess)` на
   следующей итерации делал `getWithNullSummaryForSource` с тем же LIMIT, получал
   снова те же самые хэши и ретраил LLM-вызовы. На 18K чанков с N=50 persistent
   failures это давало `18000/50 × 50 = 18000` лишних LLM-вызовов на одни и те же
   50 сломанных чанков — прямой перерасход бюджета SiliconFlow.

2. Неатомарная запись: `updateSummaries` и `updateSummaryEmbeddings` выполнялись
   раздельными транзакциями. При падении процесса (SIGTERM / network / OOM) между
   ними строка застревала с `summary != NULL, summary_embedding = NULL`. Фильтр
   `summary IS NULL` больше её не подхватывал → чанк навсегда выпадал из 3-way поиска.

## Root Cause

1. Loop обходил тот же «первый N» срез через SELECT без keyset-cursor. Код неявно
   полагался на side-effect (UPDATE строки → новая iteration не выберет её). Это
   работало ТОЛЬКО для ok и skipped веток — failed пропускали этот инвариант.
   Причина — skipped рассматривался как «опознанный skip» (писался плейсхолдер),
   а failed — как «временная ошибка» (ожидается, что retry в providerе её решит).
   На практике HTTP 400 на ту же пару (model, input) стабилен, retry не помогает.

2. Архитектурное предположение «две последовательные транзакции ≈ одна» ошибочно
   под любым не-happy-path: signal, OOM killer, disconnection. Критерий отбора
   кандидатов завязан только на `summary IS NULL`, хотя реально «полностью
   записанной» строкой следует считать обе колонки одновременно. Метод API
   должен был изначально писать обе колонки одной UPDATE в одной транзакции.

## Solution

1. Заменили `updateSummaries` + `updateSummaryEmbeddings` единым атомарным
   `updateSummaryWithEmbedding(updates: {contentHash, summary, embedding: number[] | null})`
   с одной транзакцией на batch и UPDATE обеих колонок одним SQL.
   `embedding: null` пишет SQL NULL — нужно для плейсхолдеров (skipped/failed
   не должны попадать в summary-vector поиск).

2. Добавили failed-placeholder `[failed:<reason>]` по аналогии с `[skipped:<reason>]`.
   Failed rows теперь пишутся плейсхолдером с `embedding = null`. Следующая
   итерация loop их не перевыбирает (фильтр `summary IS NULL` уже не срабатывает),
   но 3-way поиск их корректно игнорирует (фильтр `summary_embedding IS NOT NULL`
   в `searchSummaryVector` и partial HNSW-индексе).

3. Вынесли обработку батча в helper `src/commands/_helpers/summarize-batch.ts` —
   отдельная testable функция `processSummarizeBatch`. Это соответствует правилу
   RULES.md «commands/ — тонкие адаптеры» (которое нарушалось, так как вся
   бизнес-логика жила в `.action()` коллбэке). Добавили 7 regression-тестов,
   покрывающих оба бага и edge cases (empty batch, full-fail batch, mixed).

## Prevention

- **Инвариант «не перевыбирать без записи»**: любой backfill-loop, завязанный на
  SELECT-then-UPDATE и `WHERE <col> IS NULL`, должен писать какой-то маркер
  (реальное значение ИЛИ sentinel) **для каждой** обработанной строки, иначе
  loop становится retry-пулемётом при стабильных failures. Добавить чеклист-пункт
  в aif-review: «Backfill CLI-команды: все возможные исходы (ok, skip, fail)
  пишут строку-маркер?»

- **Атомарность пары колонок**: если две колонки концептуально одна запись
  (summary + summary_embedding, file + content_hash, etc.), их storage-API
  должно принимать оба параметра одной функцией и писать одной транзакцией.
  Раздельные функции допустимы только если каждая — полноценно консистентное
  состояние. Code-review red flag: два подряд UPDATE на одну и ту же строку
  из одного CLI-пути.

- **Тесты**: regression-тест должен проверять не «мой positive-case работает»,
  а «процесс, падающий в любой точке, оставляет БД в состоянии, с которого
  можно продолжить». Здесь — тест на «весь батч failed» как proxy для
  «процесс упал сразу после записи summary, но до записи embedding».

- **Архитектура CLI-команд**: бизнес-логика не должна жить в `.action()`
  Commander-коллбэках — вытаскивать в `commands/_helpers/*.ts` как testable
  функцию с явной сигнатурой. RULES.md правило «commands/ — тонкие адаптеры»
  должно автоматически включаться в aif-review / aif-plan чеклисты.
