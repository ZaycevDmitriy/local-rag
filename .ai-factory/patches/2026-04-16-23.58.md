**Severity:** medium

## Problem

1. `rag export --dry-run` печатал `undefined фрагментов` в списке источников и
   `Всего: ... NaN фрагментов` в итоговой строке. Та же проблема проявлялась и
   в интерактивном `checkbox`.
2. MCP-инструмент `search` молча игнорировал параметр `sourceName` — Zod-схема
   MCP SDK отбрасывает неизвестные поля, поэтому `search({ sourceName: 'karipos' })`
   попадал в пайплайн без фильтра, и поиск шёл по всем `active_view_id` сразу.
   Клиент (Claude Code/Cursor) не получал никакого сигнала об ошибке.

## Root Cause

1. Миграция 005 (`branch_views_rebuild`) вынесла счётчик чанков из таблицы
   `sources` в `source_views.chunk_count`. Поле `SourceRow.chunk_count` осталось
   в TypeScript-типах как `@deprecated` только для совместимости компиляции,
   но `SourceStorage.getAll()` делает `SELECT * FROM sources`, где такой колонки
   уже нет → в рантайме это `undefined`, а `total += undefined` → `NaN`.
   Команда `export-cmd` продолжала читать `source.chunk_count` в двух местах:
   интерактивный список и dry-run сводка. Реальный экспорт считал через
   `COUNT(*) FROM chunks` по всем `source_views` источника и работал корректно —
   то есть dry-run расходился с реальностью.
2. API MCP-инструментов был несимметричным: `read_source` принимает `sourceName`,
   а `search` — только `sourceId: z.string().uuid()`. Документация (`CLAUDE.md`)
   показывает пример со `sourceName`, но схема его не описывала. В результате
   параметр беззвучно отбрасывался, а в `SearchCoordinator.resolveSearchFilters`
   отсутствие `sourceId` трактовалось как «без фильтра», и `for` по `sources`
   собирал все `active_view_id`.

## Solution

1. Добавлен helper `src/commands/_helpers/chunk-count.ts::sumChunksForSource(sourceViewStorage, sourceId)` —
   чистая функция, суммирующая `chunk_count` по всем `source_views` источника.
   В `export-cmd.ts` до открытия меню считается `Map<sourceId → chunks>` через
   `Promise.all`, и оба места (checkbox и dry-run) используют этот `Map` вместо
   устаревшего `source.chunk_count`. Агрегат берётся по всем views, чтобы число
   совпадало с реальным экспортом (`exporter.ts` экспортирует все views).
2. В `SearchQuery` добавлено опциональное поле `sourceName`. В
   `SearchCoordinator.resolveSearchFilters` добавлен приватный шаг
   `resolveSourceNameFilter(query)`: он резолвит `sourceName` в `sourceId` через
   `SourceStorage.getByName`, бросает `Error('Provide either sourceId or sourceName, not both')`
   при конфликте, `Error('Source "X" not found')` при неизвестном имени.
   В `search.ts` (MCP-инструмент) поле `sourceName: z.string().optional()`
   добавлено в `inputSchema` и проброшено в `coordinator.search(...)`. Ошибки
   coordinator'а перехватываются существующим `catch`-блоком и превращаются в
   `{ isError: true, content: [{ text: 'Search error: ...' }] }`.
3. Добавлены тесты: (а) helper `sumChunksForSource` (4 кейса); (б) branch-aware
   ветка coordinator — резолв sourceName, работа с branch-параметром, ошибки
   на неизвестном имени и конфликте, fallback на все active views (5 кейсов);
   (в) MCP-обёртка search — регистрация sourceName в inputSchema, проброс в
   coordinator, корректная упаковка ошибок в `isError: true` (5 кейсов).

## Prevention

- Когда в миграции **удаляется** колонка, оставленная в типах как `@deprecated`,
  нужно grep-ом проверить все её использования в прикладном коде — тип не
  ловит `undefined`, потому что поле присутствует в интерфейсе. Либо удалять
  поле из типа сразу, либо переводить в `chunk_count?: number` и получать
  ошибку компиляции в местах, где используется без проверки.
- Для MCP-инструментов сверять симметрию API между инструментами, работающими
  с одинаковыми сущностями (`search`/`read_source`/`list_sources` все берут
  источник — параметр должен называться одинаково). Если `CLAUDE.md` или
  документация показывает параметр — он обязан жить в `inputSchema`, иначе
  Zod молча его отбросит.
- MCP-tool обязан возвращать `isError: true` (а не пустой результат) при
  явной ошибке валидации фильтра — «молчание = успех» ломает UX агентов.
- Для любого агрегата, который должен совпадать с «настоящим» подсчётом
  (dry-run vs реальный экспорт), оба пути брать из одного источника правды.
  Здесь это `source_views.chunk_count`; добавление helper'а сделало путь общим.
