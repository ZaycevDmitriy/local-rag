## Rules

### Include a grep-and-remove step when a plan drops a SQL column
**Source**: 2026-04-16-23.58.md
**Rule**: When the plan touches a migration that removes a column, add an explicit task: "grep every reference to `<column>` across `src/`, delete the field from all TypeScript row types, update every call site". Do not plan to leave the field as `@deprecated` — TypeScript does not catch runtime `undefined` when the field is on the interface but absent at runtime, and arithmetic like `total += row.field` silently becomes `NaN`. If staged removal is required, plan to convert the field to optional (`field?: number`) to force compile errors everywhere.

### Plan MCP tools with full inputSchema and API symmetry
**Source**: 2026-04-16-23.58.md
**Rule**: When a plan introduces or modifies an MCP tool, include explicit tasks for: (a) Zod `inputSchema` with every parameter shown in `CLAUDE.md` / `docs/` (Zod drops unknown fields silently); (b) parameter-name parity across tools that operate on the same entity — `search`, `read_source`, `list_sources` must all use `sourceName`; (c) `isError: true` handling for every validation / lookup failure. Missing any of these introduces a silent contract gap.

### Plan `--dry-run` and real path to share a single helper
**Source**: 2026-04-16-23.58.md
**Rule**: If the planned feature has both a `--dry-run` (or interactive preview) and a real execution path, design a shared helper in `src/commands/_helpers/` and call it from both sites. Reject plan drafts that compute the same aggregate twice in parallel code paths — they diverge after schema migrations.

### Plan backfill loops with the marker-for-every-outcome invariant
**Source**: 2026-04-17-16.01.md
**Rule**: For any backfill / indexing plan that uses SELECT-then-UPDATE with `WHERE <col> IS NULL`, include an invariant task: "every processed row writes a marker in all branches (ok / skip / fail), with sentinels for non-ok outcomes". Decide the sentinel format up-front — this repo uses `[skipped:<reason>]` / `[failed:<reason>]` text paired with `embedding = NULL`. Without this, persistent provider errors on the same inputs drain budget via infinite retries.

### Plan atomic writes for related column pairs
**Source**: 2026-04-17-16.01.md
**Rule**: When planning storage changes that introduce or update paired columns (e.g., `summary` + `summary_embedding`, `path` + `content_hash`), plan a single storage-API method that writes all paired columns in one transaction / single SQL `UPDATE`. Reject plans that propose two separate write methods for a logically-atomic record — a crash between them is unrecoverable in selectors that rely on `WHERE col IS NULL`.

### Plan `commands/_helpers/*.ts` extraction for every non-trivial CLI command
**Source**: 2026-04-17-16.01.md, .ai-factory/RULES.md
**Rule**: When a plan adds or modifies a CLI command beyond trivial option parsing, include a task to create / update the corresponding `src/commands/_helpers/<command>-<topic>.ts` with an explicit signature and a dedicated unit-test file. `.action()` callbacks must stay thin — this is an explicit project rule (`RULES.md`). Business logic in `.action()` is untestable and blocks the project's coverage requirement for `commands/`.

### New module requires a barrel `index.ts`
**Source**: ARCHITECTURE.md, .ai-factory/RULES.md
**Rule**: When a plan introduces a new module under `src/` (e.g., `src/summarize/`), include a task to create `index.ts` as the public barrel that re-exports the module's external API. Other modules must import only from this barrel — direct imports of internal files are forbidden by `RULES.md`.
