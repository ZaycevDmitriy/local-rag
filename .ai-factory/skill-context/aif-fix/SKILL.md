## Rules

### MCP-reachable logging: never `console.log`, always `console.error`
**Source**: 2026-04-16-22.25.md
**Rule**: When a fix adds, moves, or modifies `console.*` calls in `src/search/**`, `src/mcp/**`, `src/storage/**`, `src/embeddings/**`, use `console.error`. `console.log` in these directories writes to MCP's JSON-RPC stdout stream and breaks the protocol for Claude Code / Cursor. If the fix touches a shared file that has both CLI and MCP callers (e.g., `src/storage/chunk-contents.ts`), trace the call graph and treat any method invoked from `SearchCoordinator.search` or an MCP tool handler as MCP-reachable.

### Update stdout-discipline guard alongside the fix
**Source**: 2026-04-16-22.25.md
**Rule**: When a fix adds or moves MCP-reachable code (new file, new method, or relocated logic), update `MCP_REACHABLE_FILES` / `MCP_REACHABLE_METHODS` in `src/mcp/__tests__/stdout-discipline.test.ts`. Otherwise the guard will not cover the new path and the regression can re-enter silently on the next change.

### Deprecated columns require type removal, not just annotations
**Source**: 2026-04-16-23.58.md
**Rule**: When a bug report traces to runtime `undefined` / `NaN` from a field that was kept as `@deprecated` on a row type (e.g., `SourceRow.chunk_count`), the correct fix is to delete the field from the interface and update every call site in the same PR. Silencing at a single call site (`total += value ?? 0`) is a patch, not a fix — other usages will re-introduce the NaN.

### MCP-tool bugs: surface `isError: true`, never silent empty results
**Source**: 2026-04-16-23.58.md
**Rule**: When fixing a bug where an MCP tool produced wrong/empty results on bad input, make the tool return `{ isError: true, content: [{ text: '... error: <message>' }] }` rather than an empty successful payload. Silence is the worst failure mode for agent clients — they treat it as "no matches" and proceed with wrong context.

### Regression test must reproduce the exact failure path
**Source**: 2026-04-17-16.01.md
**Rule**: The regression test for a fix must fail without the fix and must exercise the same crash / state boundary that triggered the bug. Do not write a positive-path test that would have passed before the fix — that does not protect against re-introduction. For backfill / transactional bugs, the test shape is "entire batch failed" or "process killed between two UPDATEs", not "happy path still works".
