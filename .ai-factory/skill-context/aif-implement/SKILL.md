## Rules

### MCP-reachable logging discipline
**Source**: 2026-04-16-22.25.md
**Rule**: In `src/search/**`, `src/mcp/**`, `src/storage/**`, `src/embeddings/**` use `console.error` for all diagnostics. `console.log` in these directories corrupts MCP JSON-RPC stdout and silently breaks Claude Code / Cursor clients. When adding a method in a shared file (e.g., `src/storage/chunk-contents.ts`) that will be called from `SearchCoordinator.search` or any MCP tool handler, treat its logging sites as MCP-reachable from the first commit.

### Keep stdout-discipline guard in sync with MCP-reachable code
**Source**: 2026-04-16-22.25.md
**Rule**: When adding a new MCP-reachable file or method, update `MCP_REACHABLE_FILES` / `MCP_REACHABLE_METHODS` in `src/mcp/__tests__/stdout-discipline.test.ts` in the same commit. The test is the only runtime guard against `console.log` regressions; without updating the lists, new code paths ship unprotected.

### MCP tool inputSchema must mirror documentation and sibling tools
**Source**: 2026-04-16-23.58.md
**Rule**: For every MCP tool, every parameter shown in `CLAUDE.md` examples or `docs/` must be registered in the Zod `inputSchema`. `McpServer` and Zod silently drop unknown fields — missing schema entries cause filters like `sourceName` to be ignored without any error. Preserve parameter-name parity across tools that operate on the same entity: `search`, `read_source`, `list_sources` must all use `sourceName` (not `sourceId` in one and `sourceName` in another).

### MCP tools must return `isError: true` on filter/validation errors
**Source**: 2026-04-16-23.58.md
**Rule**: When an MCP tool encounters an explicit validation or lookup error (unknown `sourceName`, conflicting `sourceId` + `sourceName`, unsupported `branch`), wrap the error as `{ isError: true, content: [{ text: '<tool> error: <message>' }] }`. Returning an empty successful result is forbidden — agent callers interpret silence as "no matches" and proceed with wrong assumptions.

### Remove SQL columns from TS row types immediately on migration
**Source**: 2026-04-16-23.58.md
**Rule**: When a migration drops a column, delete the field from all TypeScript row types (e.g., `SourceRow.chunk_count`) in the same PR. Leaving the field as `@deprecated` does not prevent runtime `undefined` — `SELECT *` returns the actual columns, arithmetic like `total += row.chunk_count` silently yields `NaN`. If staged removal is unavoidable, mark the field optional (`field?: number`) to force compile errors at every call site.

### Backfill loops write a row marker for every outcome
**Source**: 2026-04-17-16.01.md
**Rule**: Any backfill loop that re-selects candidates with `WHERE <col> IS NULL` must write a marker for every processed row across ALL outcomes. Success writes the real value; skip and fail write sentinel placeholders (format used in this repo: `[skipped:<reason>]` / `[failed:<reason>]`). For paired-embedding columns, pair the sentinel with `embedding = NULL` so the row is excluded from vector search by partial HNSW filters. Without markers, persistent LLM/API failures on the same inputs turn the loop into a retry pulsemachine and burn the provider budget.

### Atomic writes for conceptually-paired columns
**Source**: 2026-04-17-16.01.md
**Rule**: If two columns form one consistent record (e.g., `summary` + `summary_embedding`, `path` + `content_hash`), the storage API must accept both values in a single function and commit them in one transaction / single SQL `UPDATE`. Designing two sequential `updateX` + `updateY` methods for a logically-atomic record is a bug — a crash between them leaves the row half-written and, if selectors rely on only one column being `NULL`, permanently invisible. Reuse pattern: `updateSummaryWithEmbedding({ contentHash, summary, embedding })`.

### Crash-recovery regression tests, not only positive path
**Source**: 2026-04-17-16.01.md
**Rule**: For backfill / indexer / migration code, add regression tests that assert the database is in a resumable state after a crash at each transactional boundary. Concrete proxy used in this repo: a test where the entire batch fails (`processSummarizeBatch` with all-fail input) must still mark every row so the next iteration does not re-select them. Positive-path-only tests pass even when the bug is live.

### CLI business logic lives in `commands/_helpers/*.ts`
**Source**: 2026-04-17-16.01.md, .ai-factory/RULES.md
**Rule**: Commander `.action()` callbacks must stay thin: parse options, build dependencies, delegate to a helper in `src/commands/_helpers/*.ts`. Non-trivial loops, state, error aggregation, and branching belong in the helper with an explicit signature and unit tests. This enforces the project rule "commands/ — тонкие адаптеры" and is the only way to get test coverage — `.action()` is practically untestable.

### Dry-run and real execution paths share one source of truth
**Source**: 2026-04-16-23.58.md
**Rule**: For commands with `--dry-run` or interactive previews (e.g., `export-cmd`), extract a shared helper (e.g., `src/commands/_helpers/chunk-count.ts::sumChunksForSource`) and call it from both the preview and the real execution. Do not duplicate SQL or aggregation logic between the two paths — they silently drift after schema migrations, producing output like `undefined фрагментов` or `NaN`.
